Step 01

작업 투입 — 빌드의 재료가
만들어지는 과정

Confluence에서 스펙이 확정되고 Jira 태스크가 만들어지면, 각 직군은 아래와 같이 파일을 Perforce에 올립니다(submit). 이것이 빌드에 들어가는 실제 작업의 첫 단위입니다.

모든 직군의 작업 투입 위치

프로그래머·기획자·아티스트 모두 bns_nc_souldev 스트림 하나에 submit합니다.
이 하나의 Perforce 스트림이 NC, ZNCS, ZTX 세 퍼블리셔의 공통 코드베이스입니다.

Jenkins와 StreamInfoWeb에서는 같은 스트림을 퍼블리셔별로 구별하기 위해 논리적 이름을 붙여 사용합니다. 세 이름은 동일한 Perforce 경로를 가리킵니다.

Jenkins/StreamInfoWeb 논리명실제 Perforce 경로담당 퍼블리셔
bns_nc_souldev//stream/bns_nc_souldevNC Korea
bns_zncs_souldev//stream/bns_nc_souldevZNCS (한국 NEO)
bns_ztx_souldev//stream/bns_nc_souldevZTX (중국)
경로직군파일 종류설명
.../Program/client/프로그래머.cpp, .h게임 클라이언트 로직
.../Program/Server/프로그래머.cpp, .h게임 서버 로직
.../Program/definition/프로그래머.xml, .dtd게임 데이터 구조(스키마) 정의

코드와 데이터가 같은 스트림 안에 있으므로, 배포 버전 스트림이 분기될 때 코드와 데이터가 함께 묶입니다. 이것이 Jenkins 5개 빌드 카테고리가 같은 폴더 아래에 있는 이유입니다.

소스 관리 구조 — Perforce 스트림

Perforce의 스트림(Stream)은 Git의 브랜치와 비슷한 개념입니다. 다른 점은 스트림이 계층 구조(parent-child)를 가지며, 하위 스트림의 변경사항을 상위 스트림으로 올리는 것을 업스트림(upstream)이라 합니다.

스트림을 이해할 때 두 가지 개념이 헷갈릴 수 있습니다. 복제 출처(어느 스트림을 원본으로 복사했는가)와 Perforce parent(변경사항이 어느 방향으로 전파되는가)는 방향이 반대입니다.

개념의미확인 위치
복제 출처어느 스트림을 원본으로 복사해서 만들었는가StreamInfoWeb parent_name 필드
Perforce parent이 스트림의 변경사항이 어느 스트림 방향으로 전파되는가Perforce p4 stream -o

예: 10200 → 10201 순서로 복제됐지만, Perforce에서는 10201의 parent = 10202, 10202의 parent = 10203, 10203의 parent = souldev 방향으로 변경사항이 전파됩니다.

스트림 계층 구조 — 복제 출처 기준

//stream/bns_nc_souldev  (NC·ZNCS·ZTX 공통 메인라인)
│
│  ① 한 달에 한 번, souldev에서 복제 → 월 메인 스트림 생성
│
├─ bns_zncs_10200  [월 메인 / souldev 복제 / 현재 Retired]
│   │
│   │  ② 그 다음 주마다 이전 스트림에서 복제 → 주간 스트림 생성
│   │
│   ├─ bns_zncs_10201  [주간 1 / 10200 복제] ← QA_3_Live (현행 라이브)
│   │   └─ bns_zncs_10202  [주간 2 / 10201 복제] ← QA_2_NextLive
│   │       └─ bns_zncs_10203  [주간 3 / 10202 복제] ← QA_1_RESERVE (최신)
│   │
│   └─ (다음 달: bns_zncs_10210 생성 → souldev에서 재복제)
│
└─ bns_nc_feature_XXXXX  [임시 피처 스트림 / souldev 복제]
        기능 개발 완료 후 souldev로 머지, 이후 Retire

※ Perforce upstream — 변경사항 전파 방향 (복제 방향과 반대):
   bns_zncs_10201 → 10202 → 10203 → bns_nc_souldev

스트림 번호 체계

끝자리 0 월 메인 스트림 — bns_zncs_10200, bns_zncs_10210 처럼 souldev에서 한 달에 한 번 복제. 다음 달 업데이트 시 10단위 순번이 올라감
끝자리 1~N 주간 스트림 — bns_zncs_10201, 10202, 10203 처럼 이전 스트림에서 주 단위로 복제
ZNCG ZNCS 월 메인 기반 복제 — ZNCG는 ZNCS 월 메인 스트림을 기반으로 다음 달에 분기 생성. 퍼블리셔마다 분기 시점이 달라 현재 10141(NeoNCT·NCA) / 10131(NeoTX) / 10111(NeoINV)으로 각기 다른 버전 사용
feature 임시 피처 스트림 — 규모가 큰 기능은 bns_nc_feature_20347, 20354~20360처럼 별도 스트림에서 작업. 완료 후 souldev 머지 → Retire

Step 02

빌드 — Jenkins

Jenkins는 2011년 Oracle이 관리하던 Hudson 프로젝트에서 fork된 오픈소스 자동화 서버입니다. 원래는 "코드가 올라오면 자동으로 빌드·테스트·배포"하기 위해 만들어졌지만, BNS에서는 이와 다른 방식으로 사용합니다.

항목Jenkins 원래 목적BNS에서의 실제 사용
빌드 트리거코드 커밋 시 자동 실행타이머 트리거 (매일 02:00 등). 커밋과 무관
빌드 대상 결정Job 설정에 하드코딩StreamInfoWeb API 실시간 조회로 동적 결정
파이프라인 스크립트Jenkins 내부에 저장Perforce에 저장, 실행 시마다 최신본 로드
노드(Agent) 역할빌드 전용 서버빌드 서버 + 실제 게임 서버까지 통합 관리
배포(CD)산출물을 서버로 전송GameServer_bl* 노드가 실제 게임 서버이며, 배포 Job이 그 위에서 직접 실행
테스트자동 단위·통합 테스트자동 테스트 없음. CheckerWeb이 사후 유효성 검증 담당
변경 감지변경 있을 때만 실행타이머로 실행하되, Perforce 변경사항 없으면 "Build is Skipped"

Jenkins가 빌드를 결정하는 원리 — StreamInfoWeb

Jenkins는 빌드 대상과 설정을 스스로 하드코딩하지 않습니다. 매 빌드마다 StreamInfoWeb을 조회하여 "무엇을, 어떻게 빌드할지"를 동적으로 결정합니다. StreamInfoWeb에서 각 스트림의 mang 값이 바뀌면, Jenkins Job 설정을 건드리지 않아도 자동으로 빌드 대상이 바뀝니다.

StreamInfoWeb API용도
GET /api/v1/stream_list전체 스트림 목록 + 각 스트림의 배포 망(mang) 할당 정보 조회
GET /api/v1/stream_detail?stream={name}특정 스트림의 퍼블리셔 설정 (게임코드, 서버주소, App ID 등) 조회
스트림mang (배포 망)의미

빌드 전체 실행 순서

① 스케줄러 발화 — 매일 02:00 타이머. _Schedule/Cooking_* Job이 StreamInfoWeb stream_list API를 조회하여, 현재 해당 mang에 할당된 스트림 목록을 확정하고 개별 빌드를 트리거
② 개별 빌드 실행 — stream_detail API 조회로 퍼블리셔 설정 획득 → Perforce에서 소스 sync → 변경사항 없으면 "Build is Skipped" → 변경사항 있으면 빌드 실행 → 산출물 Perforce submit → ReleaseWeb 등록 → 완료 메일 발송
③ Pack_Product_Stream (08:00경) — ReleaseWeb API로 최신 컴포넌트 버전 조회. ProgramBinary·GameData·GraphicData 세 컴포넌트를 묶어 Product 패키지 생성 → ProductVer 확정 (예: 8.20.20000.2506). 모두 동일하면 "Not Changed"로 건너뜀
④ CheckerWeb 레코드 등록 — Product 패키지 생성 완료 시 CheckerWeb에 자동 등록 (State=InProgress) → 데이터 무결성·실행파일 기동·게임데이터 자동 유효성 검사 실행
⑤ State=Complete 업데이트 — 검증 완료 → 배포 담당자가 결과 확인 후 배포 단계로 진행

빌드 폴더 구조 — 5개 카테고리

Jenkins 폴더명은 Perforce 스트림 논리명을 그대로 사용합니다. 각 스트림 폴더 아래 5개 빌드 카테고리 폴더가 있고, 그 안에 실제 빌드 Job들이 세분화되어 있습니다.

Step 03

빌드 검증 — CheckerWeb

CheckerWeb은 빌드 유효성 검증 결과를 퍼블리셔·버전별로 관리하는 시스템입니다. Daily/Pack_Product_Stream이 Product 패키지를 생성하면 자동으로 레코드가 등록되고, 검증이 완료되면 배포 판정을 내립니다.

동작 흐름

01 레코드 등록 — Pack_Product_Stream이 Product 패키지를 생성하면 CheckerWeb에 자동 레코드 등록 (State=InProgress)
02 자동 유효성 검사 실행 — 데이터 무결성 확인(InValid), 실행파일 기동 테스트(Crash), 게임데이터 검증(GamedState), 랭킹데이터 검증(RankingdState)
03 State=Complete 업데이트 — 배포 담당자가 결과 확인 후 배포 단계로 진행. 배포 Job 실행 시 ChangeToIng API를 호출하여 레코드를 "배포 진행 중" 상태로 표시
