02

스트림의 구조 : Souldev

Souldev로부터 파생되는 스트림

모든 작업자는 souldev라는 코드베이스에 공통적으로 작업물들을 제출한다.

Souldev 안에는 한국 본서버(NC), 한국 NEO(ZNCS), 중국 특화(ZTX) 세 가지가 공존한다.

한 달에 한 번, souldev로부터 각각의 릴리즈용 스트림이 생성된다.

규모가 큰 기능은 별도의 임시 스트림(피처 스트림)에서 진행한다. 기능이 완성되면 souldev에 합쳐지고, 피처 스트림은 삭제된다.

ZTX는 ZNCG에 포함된 중국 NEO와는 별개 항목이다.

03

스트림의 구조 : 주간 분기와 글로벌 파생

주간 업데이트를 위한 스트림을 생성하고, 해외용 파생 스트림을 생성한다

ZNCS는 릴리즈 스트림을 기준으로, 매주 주간 업데이트용 신규 스트림을 생성한다.

ZNCG는 서비스 지역(대만, 북미·유럽, 중국, 러시아)에 따라 분기 시점이 달라 각기 다른 버전을 사용한다.

04

빌드 : Jenkins

게임 서비스 전체 인프라의 오케스트레이터

Jenkins 원래 목적
코드 커밋 즉시
자동 빌드·배포
2011년 Hudson에서 fork된 오픈소스 자동화 서버. Java 기반, 수천 개의 플러그인 생태계. 코드 커밋 이벤트가 발생하면 빌드·테스트·배포가 자동으로 이어지는 CI/CD를 위해 만들어졌다.
이 환경에서의 역할
빌드·패키징·배포를
통합하는 오케스트레이터
단순 CI 도구가 아니다. 빌드 전용 서버부터 실제 게임 서버까지 모두 Jenkins Agent로 연결되어 있으며, 빌드·패키지 관리·배포가 하나의 체계에서 실행된다.

Jenkins 핵심 구성요소

구성요소역할
Controller (Master)Job 스케줄링, Agent 관리, 빌드 이력 보관. Jenkins 시스템 전체를 조율하는 마스터 노드.
Agent (Node)Controller로부터 Job을 받아 빌드·배포 작업을 직접 수행하는 서버. 역할별로 전용 Agent가 분리된다.
Job / Pipeline무엇을 어떻게 실행할지 정의한 작업 단위. Groovy 언어 기반 파이프라인 스크립트로 선언된다.
TriggerJob 실행 시점을 결정. 코드 커밋, 타이머(정해진 시각), 수동 실행, 다른 Job 완료 후 연쇄 실행 등을 지원한다.
Executor한 Agent에서 동시에 처리 가능한 Job 수. Executor 수에 따라 한 서버에서 병렬 처리 가능한 양이 달라진다.

이 환경에서의 특화 운영 방식

⏱
타이머 트리거 + 변경 감지 코드 커밋이 아닌 정해진 시간에 자동 발화, 또는 담당자가 수동 실행. 단, Perforce에서 변경사항이 없으면 빌드를 건너뛴다.
🔗
StreamInfoWeb으로 빌드 대상 동적 결정 Job에 빌드 대상을 하드코딩하지 않는다. 매 빌드 시마다 별도 관리 시스템 API를 실시간 조회하여 대상 스트림과 설정을 그 자리에서 결정한다.
📁
파이프라인 스크립트를 Perforce에 저장 Groovy 스크립트를 Jenkins 내부가 아닌 Perforce 저장소에 저장한다. 빌드 실행 시마다 최신 스크립트를 가져오므로 수정 사항이 즉시 모든 빌드에 반영된다.
🖥
실제 게임 서버가 Jenkins Agent Jenkins에 연결된 Agent 중 상당수가 실제로 게임을 서비스하는 서버다. 배포 Job이 해당 게임 서버 위에서 직접 실행되어 업데이트가 이루어진다.

5개 빌드 카테고리

각 스트림 폴더 아래 5개 카테고리 폴더가 있고, 그 안에 더 세분화된 개별 Job들이 존재한다.

카테고리역할주요 내용
Binary클라이언트 실행파일 컴파일게임 메인 실행파일, 플랫폼 바이너리 등
BinaryServer서버 실행파일 컴파일게임 서버를 구성하는 각 데몬 종류마다 독립된 Job
Definition데이터 구조 스키마 변환기획 데이터의 구조 정의 파일을 바이너리로 변환
GameData게임 수치 데이터 패킹캐릭터·아이템·퀘스트·맵 등 종류별로 분리하여 병렬 빌드
Graphic그래픽 에셋 쿠킹3D 모델·이펙트·UI 에셋을 언리얼 엔진 형식으로 변환

빌드 전체 실행 흐름

①
스케줄러 Job 발화 타이머(매일 정해진 시간) 또는 담당자 수동 실행. 빌드 관리 시스템 API를 조회하여 현재 망에 할당된 스트림 목록을 확인하고, 해당 스트림의 빌드 Job을 트리거한다.
②
개별 빌드 Job 실행 Perforce에서 최신 소스를 동기화하고 변경사항을 감지한다. 변경사항 없으면 Build Skipped 종료, 있으면 빌드를 실행하고 산출물을 패키지 저장소(ReleaseWeb)에 등록한다.
Perforce sync → 변경 감지 → 스킵 or 빌드 → ReleaseWeb 등록
③
Product 패키지 생성 (정기 실행) ReleaseWeb에서 바이너리·게임데이터·그래픽 컴포넌트의 최신 버전을 조회한다. 하나라도 새 버전이 생겼으면 세 컴포넌트를 묶어 신규 Product 패키지를 생성한다. 모두 동일하면 건너뛴다.
ReleaseWeb 조회 → 변경 감지 → 스킵 or Product 생성
④
CheckerWeb 자동 등록 Product 생성이 완료되면 즉시 CheckerWeb에 검증 레코드가 자동 등록되고 4가지 자동 검사가 시작된다. 검증이 통과되어야 다음 배포 단계로 이행할 수 있다.
Product 생성 → CheckerWeb 레코드 자동 등록 → 자동 검사 시작

05

빌드 : 패키지와 프로덕트

모듈에서 패키지로, 패키지에서 Product로

빌드된 모듈들은 그대로 쓰이지 않고, 다시 3개의 패키지로 묶인다: 그래픽 패키지, 게임데이터 패키지, 프로그램바이너리 패키지.

각 패키지는 별도 저장소에 버전별로 등록된다. 이전 패키지와 비교해 변경된 내용이 있을 때만 새 패키지가 만들어진다.

3개 패키지가 모두 모이면 하나의 Product(완성품)로 합쳐진다. Product 단위가 곧 QA망 업데이트와 라이브 배포에 쓰이는 최종 산출물이다.

빌드 검증 : CheckerWeb

패키지 정합성 자동 검증 시스템

검증은 패키지로 묶이는 시점에 시작된다. 모듈 빌드 단계에서는 검증이 없다.

패키지가 새로 만들어지면 자동으로 검증 레코드가 등록되고 4가지 항목을 자동으로 점검한다.

검증 레코드에는 실행파일 버전 / 데이터 버전 / Product 버전이 각각 따로 기록된다. 버전만 보면 어느 스트림에서 빌드된 것인지 추적 가능하다.

ZNCG는 퍼블리셔마다 다른 스트림을 사용하므로, 퍼블리셔별로 독립된 검증 레코드가 생성된다. 검증도 지역별로 각각 진행된다.
