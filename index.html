Step 01 — 소스 관리

Perforce 스트림 구조,
직접 정리하며 공부했습니다

처음에는 git의 브랜치 개념으로 접근했지만, upstream 방향이 복제 출처와 반대여서 혼동이 있었습니다.
스트림 번호 체계와 parent 개념을 직접 도식화하며 이해해 나갔습니다.

🤔 처음 접근

git 브랜치 개념으로 접근했지만, upstream 방향이 복제 출처와 반대여서 "어느 스트림이 parent이고, 변경사항이 어느 방향으로 흐르는가"를 파악하기 어려웠습니다.

📘 공부한 결과

끝자리 0 = 월 메인 스트림(souldev에서 복제), 끝자리 1~N = 주간 스트림. parent는 upstream 머지 방향(변경사항이 전파되는 방향)임을 파악했습니다.

→ 스트림 번호만 봐도 계층 구조가 읽힘

월 souldev에서 복제 → bns_zncs_10200 (월 메인 스트림) 생성
주 이전 스트림에서 복제 → 10201 → 10202 → 10203 (주간 스트림). 버그 픽스는 역방향(upstream)으로 머지 전파
글로벌 ZNCS 월 메인에서 복제 → bns_zncg_XXXXX. 4개 퍼블리셔(NeoNCT·NeoNCA·NeoINV·NeoTX)가 각각 다른 스트림 사용

Step 02 — 빌드

Jenkins의 BNS 활용 방식,
배우며 정리했습니다

일반적인 CI/CD 도구로만 알고 있었던 Jenkins가,
BNS에서는 전혀 다른 방식으로 운용되고 있었습니다.

💡 알고 있던 것

코드 커밋이 트리거 → 자동 빌드·테스트·배포. 빌드 대상과 설정은 Job에 하드코딩하는 방식이라고 알고 있었습니다.

📖 공부한 결과

타이머 트리거(매일 02:00) + StreamInfoWeb API 실시간 조회로 빌드 대상을 동적 결정. 파이프라인 스크립트도 Perforce에 저장되어 매 빌드마다 최신 버전으로 로드됩니다.

→ 170개 노드, 게임 서버 자체가 Jenkins Agent

01 Binary — 클라이언트 실행파일(BNSR) 컴파일
02 BinaryServer — GameDaemon·LobbyDaemon 등 서버 데몬별 개별 Job
03 Definition — 데이터 구조 스키마(.xml/.dtd)를 바이너리로 변환
04 GameData — 기획자 작업물(캐릭터·퀘스트·아이템·존 데이터) 패킹
05 Graphic — 언리얼 에셋 쿠킹(UE4 GPU 집약 작업, 전용 고사양 머신)

Step 03 — 검증

CheckerWeb의 역할과
검증 구조를 공부했습니다

빌드 산출물이 실제로 배포 가능한지 판단하는 기준이 무엇인지,
상태 필드와 판정 흐름을 통해 파악하고자 했습니다.

State
=Complete

검증 완료 상태

0

InValid — 데이터 오류 없음

0

Crash — 기동 크래시 없음

01 Pack_Product_Stream이 Binary·GameData·Graphic을 묶어 Product 패키지 생성 → CheckerWeb에 레코드 등록 (State=InProgress)
02 자동 유효성 검사 — 데이터 무결성(InValid), 실행파일 기동 테스트(Crash), 게임·랭킹 데이터 검증(GamedState·RankingdState)
03 State=Complete 업데이트 → 담당자가 세 조건을 확인 후 배포 단계 진행

Jenkins Job이 어떤 순서로 실행되는지 공부했습니다.

SharingClientInternal

feature 또는 개발 스트림을 개발팀 공유망에 즉시 배포. 정식 버전 스트림이 아닌 개발 중인 스트림을 직접 올릴 수 있습니다.

ProductClientQA

QA 검증망
Step 03

ProductServer + Live 반영

서버 바이너리를 별도 Job으로 배포 후 Live 서버에 반영. Jira 배포 티켓으로 버전·DB 변경·담당자 이력을 관리합니다.

Live 서비스

새로운 빌드 환경을 마주하더라도, 이처럼 구조를 분해하고 하나씩 이해하는 방식으로
빠르게 파악하기 위해 노력하겠습니다.

#2

Case Study

1인 제작 게임
'파티랏슈!!' 글로벌 런칭

21살, 해외 퍼블리셔와의 글로벌 런칭 준비가 전환점이 되었습니다.
'효율적이고 리스크를 예방하는 프로세스'에 대한 고민을 하게 된 계기입니다.

Background

해외 퍼블리셔와
출시를 준비하다

<파티랏슈!!>는 99명의 동료를 수집하며 스토리를 클리어하는 '동료 수집형 RPG'로, 20장에 달하는 스토리와 탐험 요소들을 포함한 장편 RPG입니다.

3년간의 제작 후 2020년에 공개되었다가, 해외 퍼블리셔인 HIKE Inc.(당시 Crest Inc.)로부터 5개 국어 퍼블리싱 협업 제안을 받았습니다. 이에 글로벌 출시를 위한 준비를 진행하게 되었습니다.

다만 1인 제작 게임이었기 때문에 리소스가 체계적으로 정리되어 있지 않았고, 이에 정리가 필요했습니다.

Retrospective

글로벌 런칭,
그리고 회고

<파티랏슈!!>는 2022년, 한국어·영어·일본어·중국어(간체)·중국어(번체)
5개 국어로 Steam, STOVE에 출시되었습니다.

그러나 1인 제작의 한계로 데이터 정합성 검사를 파일 형식 위주로만 체크했고,
인게임 적용에 대한 QA 검수 프로세스를 상세히 설계하지 못했습니다.
결국 일본어 버전 인게임 후반부에서 데이터 오류로 인한 부정적 피드백이 발생했습니다.

'리스크를 관리할 수 있는 방법이 없었을까'라는 아쉬움이,
